The PDF viewer comes with a fine-tuned set of CSS rules to make it responsible. It automatically adapts for large and small screens.

Unfortunately, when you're embedding the PDF viewer into your web application, this CSS rules quickly start to fail. Embedding often means that the PDF viewer can't use the full width of the screen. For example, the page you're currently viewing has a side menu (unless your viewing it on a very small smartphone). As of 2019, CSS doesn't bear such a scenario in mind.

So <ngx-extended-pdf-viewer> uses a two-part strategy:

  • If <ngx-extended-pdf-viewer> uses almost the full width of the window, the standard CSS rules are used. That's the most efficient approach.
  • Otherwise, <ngx-extended-pdf-viewer> uses JavaScript to simulate the media queries.

You can override the automagical choice of the correct approach by setting the attribute [ignoreResponsiveCSS].

To see the effect, make this window smaller. If [ignoreResponsiveCSS]="false, you'll notice some errors. In particular, the scale selector is pushed down on small screens, showing up in like a hardly-visibe ghost above the PDF file. If [ignoreResponsiveCSS]="false, these errors do not show. At the time of writing, the demo isn't perfect yet. Switch to the full-width view before changing the setting to see the effect. Otherwise, you might be confused by the current, slightly buggy state of the demo.
